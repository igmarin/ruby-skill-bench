# Config System Unification — v0.1.0 (Tech Debt)

**Status:** Stub — Not Yet Planned

## Problem Statement

The skill-bench codebase has **two parallel configuration systems** that read the same `skill-bench.json` file but interpret different formats:

### System A: `Models::Config` / `Models::Provider`
- Used by: `RunnerService` → agent spawning
- Reads: `skill-bench.json` with format `{"provider": "...", "config": {"api_key": "...", "model": "..."}}`
- Generated by: `Commands::Init` (the `skill-bench init` command)
- Works: Agent execution gets correct API key via direct options passing

### System B: `Config` facade / `Config::Store`
- Used by: `Client.call` → Judge → LLM calls
- Expects format: `{"current_llm_provider": "...", "providers": {"openai": {"api_key": "...", ...}}}`
- Loaded via: hierarchical pipeline (Defaults → Home JSON → Local JSON → ENV)
- Does NOT understand the `init` command's format

### Impact

The dual-system architecture was the root cause of bugs fixed in `plans/config-and-client-fixes-v0.1.0.md`. The immediate fix (threading API keys through the pipeline) avoids the issue, but the underlying structural problem remains:

1. `skill-bench init` generates JSON that only System A understands
2. System B's `JsonLoader` silently ignores System A's format, falling back to defaults
3. Any code path using `Client.call` without explicit `provider:` and `api_key:` will fail silently

## Proposed Solution

Merge both systems into a single `Config` entity that supports both JSON formats:

1. `Config::JsonLoader` recognizes both `{"provider": "...", "config": {...}}` and `{"current_llm_provider": "...", "providers": {...}}`
2. `Models::Config` is deprecated — all code paths use the unified `Config` facade
3. `Config::Store.for_provider()` returns config that was loaded from either format
